# 出力フォーマット定義

各部門が回答を生成する際は、このファイルで定義されたフォーマットに従うこと。

---

## 共通ルール

- 回答は **結論から先に** 書く
- 手順は **番号付きステップ形式** で記載
- 重要な警告・注意事項は **【注意】** を付けて先頭に記載
- 社内固有の情報（システム名・ルール等）は `context/` フォルダを参照して反映する

---

## 障害・トラブル対応（運用監視部・UserSupport チーム）

```
## 【状況整理】
（ユーザーの報告内容を整理して記載）

## 【原因仮説】
1. （最も可能性が高い仮説）
2. （次の仮説）

## 【調査手順】
1. （まず確認すること）
2. （次に確認すること）

## 【対処手順】
1. （手順1）
2. （手順2）

## 【再発防止案】
- （監視追加・手順改善・設定変更等）

## 【エスカレーション基準】
- （このケースでは〇〇の場合に人間にエスカレーション）
```

---

## バグ修正・改修依頼（保守開発部）

```
## 【原因分析】
（技術的な原因を記載）

## 【影響範囲】
- （影響するシステム・機能・データ）

## 【修正方針】
（どこをどう直すかの方針）

## 【修正案（コードドラフト）】
（コードブロックで記載）

## 【テスト観点】
- 正常系: （テストケース）
- 異常系: （テストケース）
- 境界値: （テストケース）
```

---

## レビュー依頼（Review チーム）

```
## 【レビュー結果サマリ】
- Critical: X件 / Warning: X件 / Info: X件

## 【指摘一覧】
### [Critical] （指摘タイトル）
- 箇所: （ファイル名・行番号等）
- 問題: （何が問題か）
- 理由: （なぜ問題か）
- 改善案: （どう直すか）

### [Warning] （指摘タイトル）
（同上）

### [Info] （指摘タイトル）
（同上）

## 【総評】
（全体的な品質評価と改善の方向性）
```

---

## ドキュメント・ナレッジ作成（ドキュメント部・Knowledge チーム）

```
## 【ドキュメント種別】
（手順書 / FAQ / テンプレート / 報告書 等）

## 【対象読者】
（誰向けに書くか）

## 【本文】
（ドキュメント本体）

## 【補足・注意事項】
（記載の前提・制約・更新が必要なタイミング等）
```

---

## 申請・手続き案内（UserSupport チーム / アカウント・権限管理部）

```
## 【対応内容】
（何をするか一言で）

## 【申請・手順】
1. （手順1）
2. （手順2）

## 【必要情報・添付物】
- （申請に必要な情報一覧）

## 【対応期間の目安】
（例: 申請後〇営業日）

## 【問い合わせ先】
（担当部門・連絡先）
```
